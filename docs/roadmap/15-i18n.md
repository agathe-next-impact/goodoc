# Chantier 15 — Internationalisation (next-intl)

## But

Permettre à un tenant de proposer son site en plusieurs langues (FR par défaut, EN en V1, plus tard ES/IT/PT). Chaque page tenant rendue selon une locale détectée via préfixe URL (`/en/a-propos`) ou cookie. Les blocs templates exposent leurs strings via `next-intl`. Cas d'usage type : médecin parisien dans le 16e qui voit beaucoup de patients anglophones / expats.

## Pré-requis

- Décision produit : i18n est-il une vraie demande des early adopters ? À arbitrer avec data utilisateurs avant d'investir 1,5 j.
- `@medsite/templates` blocs n'ont pas de string hardcodée hors data (à valider — sinon refacto plus large)

## Périmètre exact

**Inclus V1 :**
- Lib `next-intl` v3.x intégrée à `apps/web`
- Locales supportées : `fr` (default), `en`
- Routing : `/en/<route>` pour anglais, racine pour français (pas de préfixe FR)
- Champ `tenants.locales: text[]` (default `['fr']`) ; un tenant active ou non l'EN
- Strings UI traduites dans `apps/web/messages/<locale>.json` (boutons, navigation, formulaires)
- Strings tenant (titres, descriptions de pages, blog) restent dans la DB sans traduction auto V1 — le praticien duplique le contenu si besoin
- Switch de langue dans le footer si tenant a ≥ 2 locales actives

**Exclus :**
- Traduction automatique du contenu praticien
- ES/IT/PT (V2)
- Plurals/format complexes (les locales V1 sont proches)

## Étapes d'implémentation

1. Décision produit OK → installer `next-intl@^3`.
2. Créer la migration DB `ALTER TABLE tenants ADD COLUMN locales text[] DEFAULT ARRAY['fr']`.
3. Refactor route group : déplacer `(tenant)` sous `[locale]/(tenant)` et adapter `getTenant()` pour ignorer le segment locale dans le slug.
4. Mettre à jour le middleware pour : (a) parser hostname, (b) si pathname commence par `/en/` et tenant a `en` activé → forward, sinon redirect vers FR.
5. Extraire toutes les strings UI hardcodées des composants `apps/web/src/components/` et `packages/templates/src/blocks/` vers `messages/fr.json`.
6. Traduire en EN.
7. Composant `<LocaleSwitch>` rendu uniquement si `tenant.locales.length > 1`.

## Critères de done

- [ ] Tenant avec `locales=['fr','en']` accessible sur `/` et `/en/`
- [ ] Switch de langue préserve la route courante
- [ ] Tenant avec `locales=['fr']` : tout `/en/*` redirige vers `/`
- [ ] `<html lang="…">` reflète la locale active
- [ ] hreflang dans le head pour les locales actives
- [ ] `pnpm typecheck && pnpm lint && pnpm test` verts
- [ ] E2E : navigation FR ↔ EN sur tenant bilingue

## Risques connus

- Refacto route group peut casser pas mal de tests E2E existants — prévoir une journée tampon.
- Middleware avec deux dimensions (tenant + locale) devient complexe → couvrir par tests unitaires de la fonction de classification.
- Strings dans `@medsite/templates` rendent le package dépendant de `next-intl` → préférer passer les strings en props depuis `apps/web` pour garder le package découplé.

## Tests à ajouter

- Unitaires : middleware classification avec couples (host, path) variés
- E2E : tenant bilingue accessible dans 2 langues, switch fonctionnel, hreflang correct
